# Contributing to Swiq

First off, thank you for checking out Swiq! Building a language with a custom syntax tree and dedicated execution states (`reset`, `archive`, `restore`) is a massive undertaking, and community contributions are highly welcome.

By contributing to this project, you agree to keep the codebase clean, robust, and well-tested.

---

## How Can I Contribute?

### 1. Reporting Bugs
If you find a bug in the Lexer, Parser, or Interpreter:
* Check the open issues to ensure it hasn't already been reported.
* Open a new issue containing a clear description, the version number, and a minimal `.swiq` code example that breaks the interpreter.
* Include the exact `runtime_error` string generated by the C++ engine.

### 2. Suggesting Features / Syntax Modifications
Swiq has very deliberate language design choices (such as explicit function closure captures and tagged object layouts). If you want to propose changes to the syntax:
* Open an issue or discussion outlining the proposed grammar.
* Show an example of how the new syntax looks in a `.swiq` script.
* Explain how the `Lexer` tokens and `Parser` AST components would need to adapt.

### 3. Submitting Pull Requests
Ready to contribute code? Follow these steps:
1. **Fork** the repository and create your branch from `main`.
2. Ensure your code compiles cleanly with standard C++ compilation flags (supporting `std::variant`, `std::unique_ptr`, and modern STL structures).
3. If you fix a bug or add a builtin function, provide a test script in the test suite to verify it works.
4. Open a Pull Request with a clear description of what changed.

---

## Structural Code Styles

To keep the C++ codebase maintainable, please adhere to the following layout patterns:

* **Token Processing:** All new syntax rules must start with an explicit `TokenType` designation added to `lexer.h` before being scanned in `lexer.cpp`.
* **Memory Management:** Use smart pointers (`std::unique_ptr` for AST expressions/statements and `std::shared_ptr` for managed objects like `ArrayObject` or `TypedObject`) to prevent memory leaks in the runtime engine.
* **Error Handling:** When a rule is violated, throw a `std::runtime_error` complete with line numbers fetched from the token metadata:
  ```cpp
  throw std::runtime_error("Interpreter error at line " + std::to_string(line) + ": message");
  ```

## Development Workflow Hint

If you are developing or testing your build on mobile environments via Termux, ensure your local build toolchain updates the output binary destination correctly so you don't accidental run cached interpreter builds during debugging loops!
