# Contributing to Swiq

First of all, thank you for considering contributing to Swiq!

Whether you're fixing bugs, improving documentation, suggesting features, or writing code, every contribution helps.

---

## Before You Start

Please:

- Read this document completely.
- Follow the Code of Conduct.
- Check if an issue already exists before creating a new one.

---

## Reporting Bugs

When reporting a bug, please include:

- Swiq version
- Operating System
- Steps to reproduce the problem
- Expected behaviour
- Actual behaviour
- Screenshots or logs if applicable

The more information you provide, the easier it is to fix.

---

## Suggesting Features

Feature requests are welcome.

Please explain:

- What problem it solves.
- Why it would benefit Swiq.
- Any ideas for implementation (optional).

---

## Contributing Code

1. Fork the repository.
2. Create a new branch.

```bash
git checkout -b feature/my-feature
```

3. Make your changes.
4. Test your changes.
5. Commit with a descriptive message.

Example:

```text
Add parser support for switch expressions
```

6. Push your branch.
7. Open a Pull Request.

---

## Coding Style

Please try to keep code:

- Readable
- Consistent
- Well documented where necessary
- Free of unnecessary complexity

If you're modifying parser or interpreter code, include comments where the logic isn't immediately obvious.

---

## Documentation

Documentation improvements are always appreciated.

This includes:

- Fixing grammar
- Explaining features
- Adding examples
- Improving formatting

---

## Pull Requests

Before opening a Pull Request, please make sure:

- The project builds successfully.
- Your changes don't introduce obvious bugs.
- Documentation is updated if needed.

---

## Questions

If you're unsure about something, feel free to open an issue first before beginning major work.

Thank you for helping improve Swiq!
